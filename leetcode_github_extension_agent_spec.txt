PROJECT SPEC / AGENT PROMPT
LeetCode Accepted Solutions Auto-Push Chrome Extension + FastAPI Backend

Purpose
Build an automation where users sign up with GitHub, install/use a Chrome extension, solve LeetCode problems normally, and whenever their own LeetCode submission becomes Accepted, the system sends the accepted solution to a FastAPI backend. The backend generates an LLM explanation and pushes the solution plus explanation into a single GitHub repository owned by that user.

This file is intended to be read by coding agents such as Codex, Cursor, Claude Code, or similar tools. Follow it as the source of truth.

====================================================================
1. CORE PRODUCT IDEA
====================================================================

Build a Chrome-only extension and FastAPI backend.

User journey:
1. User installs the Chrome extension.
2. User opens the extension popup.
3. User must sign up/login with GitHub before using the extension.
4. GitHub OAuth redirects to the FastAPI backend callback URL.
5. Backend creates/saves the user in PostgreSQL.
6. Backend stores GitHub authorization details securely.
7. User completes onboarding.
8. User solves LeetCode problems manually in the browser.
9. Extension detects when a submission result becomes Accepted.
10. Extension sends the accepted code/problem metadata to FastAPI backend.
11. Backend generates an LLM explanation using LangChain with either OpenAI or Gemini, configurable through .env.
12. Backend creates one GitHub repository named something like "leetcode-problems" or "LeetCode-Problems" if it does not already exist.
13. Backend pushes the solution code, explanation, and metadata into that single repository.

The automation goal:
Accepted LeetCode solution -> backend -> LLM explanation -> GitHub commit.

3. IMPORTANT SAFETY / COMPLIANCE REQUIREMENTS
====================================================================

The extension must not behave in a way that risks abuse, evasion, or account flagging.

Do:
- Only operate on LeetCode pages required for the feature, such as leetcode.com/problems/* and relevant submission/result pages.
- Only capture the signed-in user's own manually submitted solution.
- Prefer DOM observation and normal page-state detection where reliable.
- If observing LeetCode network/API behavior is needed, keep it passive and minimal.
- Never modify LeetCode requests or responses.
- Never automate solving or submitting problems.
- Never bypass CAPTCHA, anti-bot checks, rate limits, authentication flows, or LeetCode protections.
- Never scrape aggressively.
- Never collect LeetCode cookies, passwords, session tokens, or unrelated browsing data.
- Rate-limit backend calls and deduplicate accepted submissions.
- Ask for only required Chrome permissions.

Do not implement stealth, anti-detection, botting, account-flag bypass, or any technique intended to evade LeetCode platform protections.

Interpret the requirement "do not get the user flagged" as:
- Make the extension privacy-preserving.
- Avoid excessive requests.
- Avoid automated submissions.
- Avoid modifying LeetCode behavior.
- Capture only user-owned accepted submissions after normal user action.

8. BACKEND API REQUIREMENTS
====================================================================

Base path: /api/v1

Auth endpoints:

GET /api/v1/auth/github/login
- Redirects user to GitHub OAuth authorization URL.

GET /api/v1/auth/github/callback
- Receives code/state from GitHub.
- Exchanges code on backend.
- Creates/updates user.
- Creates local session/JWT.
- Redirects back to extension success page or returns token using a safe extension-compatible flow.

GET /api/v1/me
- Requires backend session token.
- Returns current user info and GitHub connection status.

POST /api/v1/auth/logout
- Revokes local session.

Submission endpoints:

POST /api/v1/submissions/leetcode
- Requires authenticated user.
- Called by Chrome extension after accepted LeetCode submission.
- Validates payload.
- Deduplicates by user_id + problem_slug + language + code_hash.
- Stores submission.
- Generates LLM explanation.
- Pushes code/explanation to GitHub.
- Returns sync status.

GET /api/v1/submissions
- Optional for debugging/admin/user history.

GitHub endpoints:

POST /api/v1/github/ensure-repo
- Requires authenticated user.
- Creates single repo if missing.
- Saves repo info in DB.

POST /api/v1/github/sync/{submission_id}
- Optional manual retry endpoint.
- Pushes a stored accepted submission to GitHub.

Health endpoint:

GET /health
- Returns backend health.

15. SUBMISSION PROCESSING FLOW
====================================================================

When extension detects Accepted:

1. Extract payload:
   - problem_slug
   - problem_title
   - difficulty if known
   - language
   - code
   - status
   - leetcode_url

2. Send to:
   POST /api/v1/submissions/leetcode

3. Backend validates:
   - user session
   - status must be Accepted
   - code must not be empty
   - problem_slug/title must exist

4. Backend computes:
   - code_hash = SHA256(code + language + problem_slug)

5. Backend checks duplicate:
   - if same user/problem/language/code_hash already pushed, return already_synced.

6. Backend stores submission.

7. Backend generates explanation through configured LLM.

8. Backend ensures GitHub repo exists.

9. Backend pushes:
   - solution file
   - explanation.md
   - metadata.json

10. Backend updates submission with commit SHA and file paths.

11. Extension displays sync success.

====================================================================
16. ERROR HANDLING REQUIREMENTS
====================================================================

Handle these cases cleanly:
- User not logged in.
- GitHub authorization failed.
- GitHub token expired/revoked.
- Repo creation failed.
- GitHub push failed.
- LLM provider/key/model missing.
- LLM generation failed.
- Duplicate submission.
- LeetCode page structure changed and code extraction failed.
- Backend unavailable.

For LLM failure:
- Either retry once or push solution with a fallback explanation saying explanation generation failed.
- Do not lose the accepted code.

For GitHub failure:
- Save submission in DB with failed status so sync can be retried.
